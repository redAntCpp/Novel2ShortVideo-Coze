## 单集节点提取（代码）

```python
# 在这里，您可以通过 'args'  获取节点中的输入变量，并通过 'ret' 输出结果
# 'args' 已经被正确地注入到环境中
# 下面是一个示例，首先获取节点的全部输入参数params，其次获取其中参数名为'input'的值：
# params = args.params; 
# input = params['input'];
# 下面是一个示例，输出一个包含多种数据类型的 'ret' 对象：
# ret: Output =  { "name": '小明', "hobbies": ["看书", "旅游"] };

import json
async def main(args: Args) -> Output:
    params = args.params
    #ep_list、
    # 构建输出对象
    ep_content = extract_episode_content(json.loads(params['ep_list']), params['ep_id'], params['story_text']);
    ret: Output = {
        "key0": "123", # 拼接两次入参 input 的值
        "key1": ["hello", "world"],  # 输出一个数组
        "key2": { # 输出一个Object 
            "ep": ep_content['ep'],
            "title":ep_content['title'],
            "content":ep_content['content']        
        },
    }
    return ret

def extract_episode_content(json_data, ep_id, storytext):
    target_episode = None
    
    # 1. 遍历嵌套结构查找对应的 ep
    for unit in json_data.get("story_units", []):
        for episode in unit.get("episodes", []):
            if episode.get("ep") == ep_id:
                target_episode = episode
                break
        if target_episode:
            break

    # --- 关键修复：安全卫语句 ---
    # 如果没找到，直接返回一个包含错误信息的字典，而不是字符串
    if not target_episode:
        return {
            "ep": ep_id,
            "title": "未找到",
            "content": f"错误：未找到编号为 {ep_id} 的集数内容。"
        }

    # 2. 提取锚点信息 (此时 target_episode 绝对不是 None)
    title = target_episode.get("title")
    start_anchor = target_episode.get("text_start_anchor")
    end_anchor = target_episode.get("text_end_anchor")

    # 3. 在 storytext 中定位并截取
    start_idx = storytext.find(start_anchor)
    end_idx = storytext.find(end_anchor)

    if start_idx == -1 or end_idx == -1:
        extracted_text = "警告：未能在提供的文本中匹配到完整的锚点内容。"
    else:
        extracted_text = storytext[start_idx : end_idx + len(end_anchor)]

    # 4. 组装结果
    return {
        "ep": ep_id,
        "title": title,
        "content": extracted_text
    }
```

提示词

```tex
你是一名专业影视编剧与短剧工业化生产专家，擅长将小说内容转化为“可直接用于拍摄与分镜拆解”的结构化剧本。

【输入】
- 原文区间：{{content}}
- 本集信息：{{episode_info}}
- 人物设定：{{all_character_card}}
- 时代背景：{{story_analyze.era_setting}}

【核心目标】
将该原文区间改编为一集短剧剧本，输出结构必须稳定、可控，并严格绑定原文，不得偏离剧情。
单集时长：约3分钟（±30秒）

--------------------------------------------------

【一、剧本结构要求】

必须拆分为多个场景（scene），每个场景包含：

- scene_id
- scene_name 场景名称（精简固定地点，不带状态/时间修饰，符合时代背景）
- space_attribute 空间属性（室内 / 室外等）
- scene_describe（场景描述）
- base_environment(详细描述空间结构（建筑、材质、布局）)
- time（时间：白天/夜晚/连续等）
- visual_style（视觉风格），如日光 / 暖光 / 冷光 / 烛光 / 月光
- summary（该场景的剧情作用）
- base_prompt（场景视觉锚点,用于AI绘图的“稳定Prompt”，场景固定结构，固定风格，只写永久不变的建筑/布局）
- change_dimensions（场景变化），如
  "整洁/破坏/明亮/昏暗"
- environmental_elements，环境元素（具体物件）
- scene_status（当前的场景状态）
为该场景定义常见状态，允许拓展：
[
  "平静",
  "紧张",
  "对峙",
  "混乱",
  "崩溃"
]




--------------------------------------------------

【二、人物引用规范（关键）】
characters，（出场人物，使用结构化引用）
所有人物必须使用结构化标识：

{
  "char_id": "",
  "phase_id": "",
  "state_id": "" （可选）
}

规则：
1. char_id：必须（唯一人物）
2. phase_id：必须（人物所处阶段）
3. state_id：仅在以下情况使用：
   - 上台表演 / 扮演角色
   - 易容 / 伪装
   - 精神异常 / 情绪极端变化
   - 非人状态（如鬼、附身等）
禁止滥用 state_id。

--------------------------------------------------

【三、内容结构（beats）（核心）】

每个场景必须由多个 beats 构成，每个 beat 是“最小可拍摄单位”。允许的类型：

1. dialogue（对白）
2. action（动作/行为）
3. expression（表情/情绪表现）
4. narration（旁白）

----------------------------------

【四、旁白（narration）使用规则（重要升级）】
允许使用旁白，但必须严格限制：

仅允许用于：
- 时间/空间过渡
- 背景信息补充（无法通过画面表达）
- 第一人称讲述
- 节奏压缩
- 人物评价

禁止用于：
- 直接说明人物情绪
- 重复画面已表达的信息
- 替代剧情推进

----------------------------------
【五、表演类型（performance_type）】
用于描述“如何被表演出来”（核心）
可选值：
- speaking：正常说话
- singing：唱（包括戏曲、歌曲）
- chanting：念（戏曲念白、仪式语言）
- voice_over：旁白或内心独白
- silent：无语言，仅表演

【规则】
1. 即使是 dialogue，也必须判断是：
   - speaking / singing / chanting
2. 出现以下词语必须强制判断：
   - “唱”“唱道”“开嗓” → singing
   - “念”“念白” → chanting
3. 如果发生在戏台/表演场景：
   - 押韵 → singing
   - 普通说话 → chanting（戏曲念白）
4. narration 通常对应 voice_over（如果用于旁白）
5. expression 一般为 silent
6. 优先从“表演形式”角度判断，而不是文本结构

--------------------------------------------------

【六、改编规则（极重要）】

1. 所有剧情必须来源于原文，不允许新增剧情
2. 可以压缩、重组，但不得改变事件顺序
3. 禁止编造原文不存在的冲突或反转
4. 心理描写必须转化为“动作 / 表情 / 对话”
5. 旁白不得改变原文语义

--------------------------------------------------

【七、节奏控制（短剧核心）】

1. 开头（前10秒）必须有吸引点（冲突 / 悬念 / 异常）
2. 每个场景必须推动剧情（禁止无效场景）
3. 信息呈现必须前置（避免拖沓）

4. 结尾处理：
   - 若不是最后一集：
     必须形成“钩子”（悬念 / 信息差 / 危机未解）
   - 若是最后一集：
     必须“收束剧情”，禁止新增悬念
     可保留“情绪余味”，但必须符合原文

--------------------------------------------------

【八、对白要求】

- 简短、口语化、符合时代背景
- 禁止解释性台词
- 必须符合人物语言模型（结合人物设定）

--------------------------------------------------
【九、情绪与语速控制规则（重要）】
1.每个 beat，当type类型为dialogue、 narration时，则必须包含emotion字段,字段不能为空（其他类型不用），子字段为：
- emotion_type（情绪类型）
- emotion_scale（1.0~5.0）
- emotion_control（情绪控制方式）
- speed_ratio（语速）

2. emotion_typ枚举值：
happy / sad / angry / surprised / fear / hate / excited / coldness / neutral
3. emotion_control 枚举值
- suppressed（压抑）
- neutral（正常）
- expressive（外放）
- explosive（爆发）
4. speed_ratio 规则（必须遵守）
必须结合 performance_type：
   - speaking：0.8~1.3
   - singing：0.6~1.0（偏慢）
   - chanting：0.7~1.1
   - voice_over：0.7~1.0
必须结合 emotion_control：
   - suppressed → 较慢（如 0.7~0.9）
   - expressive → 正常或略快（1.0~1.2）
   - explosive → 较快（1.2~1.5）

3. 极端值（如 2.0）除非明确需要，否则不要轻易使用极端值

【九、输出格式（必须严格遵守）】
1. 输出的必须JOSN格式，其中JSON的所有key必须是英文。
2. 对于输入的charId、charName、phaseId、phaseName,stateid等关键字段，不要改动内容。方便后续匹配。
3. JSON格式示例如下：
{
	"ep": "",
	"scenes": [{
		"scene_info": {
			"scene_id": "",
			"scene_describe": "",
			"scenename": "",
			"visual_style": "",
			"base_prompt": "",
			"time": "",
			"hange_dimensions": "",
			"scene_status": "",
			"environmental_elements": [],
			"space_attribute": ""
		},
		"characters": [{
			"char_id": "",
			"phase_id": "",
			"state_id": ""
		}],
		"summary": "",
		"beats": [{
			"type": "dialogue/action/expression/narration",
			"performance_type": "singing",
			"character": {
				"char_id": "",
				"phase_id": "",
				"state_id": ""
			},
			"content": "",
			"emotion": {}
		}]
	}],
	"ending_hook": ""
}



```
